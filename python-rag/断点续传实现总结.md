# 断点续传功能实现总结

## 问题描述

项目中的医学 RAG 评估任务（`complete_eval.py` 和 `enhanced_eval.py`）需要处理数百道 MedQA 题目，每道题都需要调用 LLM API。整个过程可能持续数小时。

**存在的问题**：
- 长时间运行时，如果遇到异常中断（网络问题、系统崩溃、手动中断等），之前完成的工作全部白费
- 重新开始需要浪费大量时间和 API 配额
- 无法从上次中断的位置继续

## 解决方案

实现了完整的断点续传功能，包括：

### 1. 核心模块：progress_manager.py

**位置**：`python-rag/app/rag/progress_manager.py`

**主要功能**：
- ✅ 自动保存评估进度（每处理完一道题立即保存）
- ✅ 加载检查点并恢复进度
- ✅ 自动检测中断的任务
- ✅ 支持多个任务独立的检查点
- ✅ 备份机制防止检查点损坏

**核心类**：
- `EvaluationProgressManager` - 进度管理器主类
- `CheckpointData` - 检查点数据结构

### 2. 评估脚本集成

#### complete_eval.py
**修改内容**：
- 导入 `EvaluationProgressManager`
- 在 `main()` 中初始化进度管理器
- 修改 `evaluate_dataset()` 函数支持断点续传参数
- 修改 `find_best_top_k()` 函数支持进度保存
- 在测试集评估时自动检测并恢复中断的任务
- 完成后自动清除检查点

**支持的恢复场景**：
- 开发集超参数搜索中断
- 测试集评估中断
- 任意题目处理失败

#### enhanced_eval.py
**修改内容**：
- 与 complete_eval.py 类似的集成
- 支持增强版评估的断点续传
- 保存额外的配置信息（混合检索、查询重写等优化标志）

### 3. 智能启动脚本：run_with_resume.py

**功能**：
- 自动检测所有中断的评估任务
- 显示中断任务的详细信息（进度、时间戳、准确率等）
- 支持交互式选择要恢复的任务
- 支持批量恢复所有中断任务
- 支持强制从头开始（`--no-resume` 参数）

**使用方式**：
```bash
# 自动检测并选择恢复
python run_with_resume.py --auto

# 运行特定脚本（自动恢复）
python run_with_resume.py complete_eval

# 强制从头开始
python run_with_resume.py complete_eval --no-resume
```

## 技术实现细节

### 检查点文件结构

**保存位置**：`results/evaluation/`

**文件命名**：
- `checkpoint_complete_eval.json` - complete_eval 的检查点

- `checkpoint.backup.json` - 备份检查点

**数据结构**：

### 保存机制

```python
# 每处理完一道题目的保存流程
for i, question in enumerate(questions, start_from + 1):
    result = evaluate_question(question)
    results.append(result)
    
    # 立即保存检查点
    progress_mgr.save_checkpoint(
        dataset_name=dataset_name,
        total_questions=len(questions),
        processed_questions=i,
        current_top_k=top_k,
        results=results,
        correct_count=correct,
        total_count=total,
        elapsed_time=elapsed,
        config=config_dict,
        script_name="complete_eval",
    )
```

### 恢复流程

```python
# 启动时检测并恢复
progress_mgr = EvaluationProgressManager(output_dir=config.OUTPUT_DIR)

# 检查是否需要恢复
resume_test = progress_mgr.should_resume(script_name="complete_eval_test")

if resume_test:
    resume_info = progress_mgr.get_resume_info(script_name="complete_eval_test")
    print(f"🔄 Resuming from question {resume_info['start_from'] + 1}")
    
    # 从上次中断的位置继续
    test_results = evaluate_dataset(
        vectorstore,
        llm_generator,
        test_set,
        top_k=best_k,
        dataset_name="Test Set",
        progress_mgr=progress_mgr,
        start_from=resume_info['start_from'],
        initial_results=resume_info['results'],
        initial_correct=resume_info['correct_count'],
        initial_total=resume_info['total_count'],
        initial_elapsed=resume_info['elapsed_time'],
    )
else:
    # 从头开始
    test_results = evaluate_dataset(...)

# 完成后清除检查点
progress_mgr.clear_checkpoint(script_name="complete_eval_test")
```

## 性能影响

| 操作 | 时间开销 | 占比 |
|------|---------|------|
| 保存检查点（每题） | ~5-10ms | <0.1% |
| 加载检查点 | ~50-100ms | 一次性 |
| 恢复流程 | ~1-2 秒 | 一次性 |

**结论**：对整体性能影响可忽略不计，但避免了长时间重跑的风险。

## 优势与收益

### 优势

1. **零损失**：中断后不会丢失已完成的进度
2. **自动恢复**：无需手动干预，自动检测并恢复
3. **透明操作**：用户无需关心实现细节
4. **多重保护**：主检查点 + 备份检查点
5. **灵活控制**：支持强制从头开始

### 收益

假设评估任务：
- 总题数：500 题
- 每题耗时：~10 秒（包括 API 调用）
- 总耗时：~5000 秒 ≈ 1.4 小时

**无断点续传**：
- 中断后重跑：浪费 1.4 小时 + API 配额

**有断点续传**：
- 中断后恢复：只处理剩余题目
- 最坏情况（499 题后中断）：只浪费 10 秒

**节省**：
- 时间：最多节省 1.4 小时
- API 配额：最多节省 500 次调用

## 注意事项

1. **检查点文件占用空间**
   - 每个检查点约几 MB
   - 任务完成后自动清除

2. **不要同时运行相同任务**
   - 可能导致检查点冲突

3. **确保磁盘空间**
   - 建议至少 100MB 可用空间

4. **网络稳定性**
   - 虽然支持断点续传，但稳定网络更高效
